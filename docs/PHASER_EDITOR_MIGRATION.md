# Phaser Editor Migration Guide

## Overview

BikeBrowser uses **Phaser 3** as the game runtime inside a React/Electron/Vite app. This document describes the migration from a fully hand-coded Phaser workflow to a **Phaser Editor-driven workflow** where scene layout, asset management, and prefabs are authored visually.

**Key principle:** Phaser remains the runtime. Phaser Editor becomes the primary scene/asset/prefab authoring tool.

## Ownership Boundaries

### Editor-Owned (will be regenerated by Phaser Editor)

| Area | Files | What it controls |
|------|-------|-----------------|
| Scene layout | `editor-scenes/*Base.js` | Static object placement, walls, floor, decorations, furniture positions, zone markers |
| Asset packs | `public/game/editor-assets/packs/*.json` | Asset loading manifests |
| Prefab definitions | `prefabs/*.js` (future: `.scene` files) | Reusable object compositions |
| Raw assets | `public/game/editor-assets/images/`, `tilemaps/` | Sprites, tilesets, images |

### Code-Owned (never touched by Phaser Editor)

| Area | Files | What it controls |
|------|-------|-----------------|
| React integration | `GamePage.jsx`, `GameContainer.jsx` | Route mounting, HUD overlay, dialog, pause, joystick |
| Phaser config | `config.js` | Scene registration, physics, scale |
| Scene logic | `scenes/GarageScene.js`, `scenes/NeighborhoodScene.js` | Player spawn, physics wiring, transitions, quests, audio, save |
| Game systems | `systems/*.js` | Save/load, quest progression, inventory |
| Game data | `data/*.js` | Quest definitions, items, shop, progression, world layout |
| Entities | `entities/Player.js`, `entities/Npc.js` | Character behavior, input, rendering |
| Audio | `audio/*.js` | Web Audio API manager, manifests, procedural sounds |
| UI utilities | `ui/*.js` | Safe zones, HUD data helpers |

## Asset Pack Organization

Asset packs use the standard Phaser Asset Pack JSON format. They are split by purpose:

| Pack | File | Purpose |
|------|------|---------|
| `preload` | `preload-pack.json` | Core assets loaded before any scene (world map) |
| `audio` | `audio-pack.json` | All music, stingers, ambient tracks |
| `garage-scene` | `garage-scene-pack.json` | Garage-specific assets (currently empty — procedural) |
| `neighborhood-scene` | `neighborhood-scene-pack.json` | Neighborhood-specific assets (world map) |
| `ui` | `ui-pack.json` | UI sprites (currently empty — React handles UI) |

### Loading asset packs in scenes

```js
import { loadPack } from '../assetPackLoader.js';

preload() {
  loadPack(this, 'preload');
  loadPack(this, 'garage-scene');
}
```

### Adding new assets

1. Drop the image/spritesheet file into `public/game/editor-assets/images/<category>/`
2. Add an entry to the appropriate pack JSON file
3. (Or) Use Phaser Editor's asset pack editor to add it visually
4. The asset is available in scenes via its key after the pack loads

## Scene Architecture: Generated vs Custom

### Pattern: Base class (editor) + Subclass (custom)

```
editor-scenes/GarageSceneBase.js    ← Editor regenerates this
        ↑ extends
scenes/GarageScene.js               ← Custom logic, never touched by editor
```

### How it works

1. **Base class** (`*SceneBase.js`) contains `editorCreate()` which places all static visual objects
2. **Subclass** (`*Scene.js`) calls `this.editorCreate()` from its own `create()` method
3. Base class exposes named references (e.g., `this.exitZone`, `this.collisionBodies`) for the subclass to wire up physics and behavior
4. The subclass handles: player spawn, physics overlaps, transitions, quest logic, save, audio, state-dependent content

### Important rules

- **Never put game logic in `editorCreate()`** — only visual composition
- **Never put layout in the subclass `create()`** — only logic that references editor-created objects
- **State-dependent objects** (upgrades, quest-complete markers) stay in the subclass since they depend on runtime game state
- When Phaser Editor regenerates the base class, the subclass is untouched

## How to Update a Scene Without Overwriting Logic

1. Open the scene in Phaser Editor
2. Make visual changes (move objects, add decorations, etc.)
3. Save and compile the scene
4. Phaser Editor regenerates the `*SceneBase.js` file
5. The custom `*Scene.js` file is untouched
6. Test in the browser — `npm run dev:web`

---

## How to Add a Prefab

### Code-based (current approach)

1. Create a new file in `src/renderer/game/prefabs/`
2. Export a class that creates a composition of Phaser objects
3. Instantiate it in the scene's `editorCreate()` or `create()`

### Editor-based (future)

1. Create a new prefab scene in Phaser Editor
2. Compose the visual objects
3. Set prefab properties (configurable per-instance)
4. Phaser Editor generates a prefab class
5. Place instances in scene files

## What Must Be Done Manually Inside Phaser Editor

These steps cannot be automated from code — they require human interaction with the Phaser Editor GUI:

1. Scene visual composition (drag-and-drop object placement)
2. Prefab scene creation and configuration
3. Asset pack visual editing (alternative to hand-editing JSON)
4. Tilemap layer setup (when tileset art is available)
5. Animation key setup from spritesheets
6. Scene compilation (File > Compile Scene)

## Manual Phaser Editor Workflow (Step-by-Step)

### First-time setup

1. Install Phaser Editor v5 (desktop app)
2. Open the BikeBrowser repo folder as a project:
   - File > Open Folder > select `C:\Users\yazan\bikebrowser`
   - Phaser Editor reads `phasereditor2d.config.json`
3. Set the Play URL to `http://localhost:5173/play`
4. Start the dev server in a terminal: `npm run dev:web`

### Working with asset packs

1. In Phaser Editor, open the Asset Pack editor
2. Navigate to `public/game/editor-assets/packs/`
3. Open the relevant pack file (e.g., `garage-scene-pack.json`)
4. Use the GUI to add image/spritesheet/audio entries
5. Point URLs to files in `public/game/editor-assets/images/` or `public/game/audio/`
6. Save the pack file

### Working with scenes

1. Open `src/renderer/game/editor-scenes/GarageSceneBase.js` in Phaser Editor
   - Note: Until `.scene` files exist, the editor reads the JS class
2. Use the Scene Editor to visually place/adjust objects
3. When done: File > Compile Scene (or Ctrl+Shift+B)
4. The editor updates `GarageSceneBase.js` with new positions
5. Test in browser: the subclass `GarageScene.js` picks up changes automatically

### Creating a new scene

1. File > New > Scene File
2. Save in `src/renderer/game/editor-scenes/`
3. Name the base class `NewSceneBase`
4. Create a matching subclass in `src/renderer/game/scenes/NewScene.js` that extends it
5. Register the scene in `config.js`

### Creating a prefab

1. File > New > Prefab Scene
2. Save in `src/renderer/game/prefabs/`
3. Compose the object visually
4. Define prefab properties for per-instance configuration
5. Compile the prefab
6. Use the prefab in scene files by adding instances

### Testing changes

1. Ensure `npm run dev:web` is running
2. Open `http://localhost:5173/play` in a browser
3. In Phaser Editor, click Play (uses the configured playUrl)
4. Verify visual changes appear correctly
5. Test game logic: walk around, interact with NPCs, complete quests
6. Check console for errors

## Guardrails

- Do NOT replace Phaser with another runtime
- Do NOT break React integration (GameContainer.jsx, HUD)
- Do NOT put game logic in editor-generated files
- Do NOT fake `.scene` file formats — use real Phaser Editor or hand-author the base classes
- Do NOT move audio to Phaser's sound system (Web Audio API approach works cross-framework)
- Do NOT regress the immersive `/play` route behavior
